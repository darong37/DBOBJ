# DBOBJ 設計変更仕様書
Date: 2026-06-12

## 概要

[design-concept.md](../../design/design-concept.md)（2026-06-12 改訂）の To Do「`in()` の追加（Table 連携のデータ投入）」「psql 連動の独立モジュール化（Psql）」「spool() の引数で確定モードを指定（ORDER BY 解析の廃止）」を具体化する。

- 独立パッケージ **DBOBJ::Psql** を新設する。psql プロセス連動（起動・テーブル一式投入）の実体はすべて Psql が持つ。パッケージは分離するが、ファイルは `src/DBOBJ.pm` 1つにまとめる（bind の配布単位を 1 ファイルに保つため）
- DBOBJ に `in($dir, $tbl)` を新設する。Table プロジェクトが生成するテーブル一式（DDL・TSV・keys ファイル）を psql 経由で DB へ一括投入し、`$self` を返す
- DBOBJ の `psql()`・`in()` は `$self->{dbname}` を渡して Psql を呼ぶ公開 API とする
- `spool($spool_id)` を `spool($spool_id, @confirm)` に変更する。Spool の確定（confirm）まで行い、確定モードは `@confirm` の形で判別する。SQL の `ORDER BY` 解析は全廃する
- 接続情報を `new()` の1箇所に集約する。`PGHOST`・`PGPORT`・`PGUSER` は内部状態（`host`・`port`・`user`）へ取り込み、Psql は DBOBJ オブジェクト経由でのみ使う（単独利用は想定しない）

## 変更の背景

- `in()` はレガシー実装（旧 DBOBJ の `sub in`）の移植であり、第二コンセプト「psql との連動」の具体化である
- DBI では扱いにくい `\copy` を psql に任せ、DDL 実行 → データ投入 → キー付与を 1 呼び出しで完結させる
- レガシーの `$bgn` / `$end` 引数（分割 TSV の範囲指定）は廃止し、単一 TSV か分割 TSV かはファイルの存在から自動判別する
- DBI でやるもの（接続・SQL 実行・取得系）と psql プロセスでやるものは責務が別であるため、psql 連動は DBOBJ の内部関数ではなく独立モジュールへ切り出す（development-policy「切り出したい処理は名前を持った独立モジュールとして外に出す」。Table → TsvUtil の前例）
- `in()` 実装後、Table プロジェクト側で `out($dbo)` の DB 結合テストを追加する（本仕様の範囲外。別セッションで実施）

## API 変更

### 新設（Psql パッケージ）

`src/DBOBJ.pm` 内に `package DBOBJ::Psql` を新設する。公開関数は次の2つで、**内部関数を持たない**。ファイルを分けないのは bind の配布単位（`lib/DBOBJ.pm` 1ファイル）を保つためで、`use DBOBJ` だけで DBOBJ::Psql もロードされる。

| 関数 | 役割 | 入力 | 出力 |
|---|---|---|---|
| `DBOBJ::Psql::run($dbo, @args)` | psql 起動（`-f`・`-c` 等の引数をそのまま渡す） | `$dbo`, `@args` | ― |
| `DBOBJ::Psql::in($dbo, $dir, $tbl)` | テーブル一式（DDL・TSV・keys）の DB 投入 | `$dbo`, `$dir`, `$tbl` | ― |

`$dbo` は接続済みの DBOBJ オブジェクト（`new($dbname)` の戻り値）。Psql の単独利用は想定しない。psql 連動が動く道筋は「`DBOBJ->new($dbname)` で接続オブジェクトを作り、それを経由する」の1本だけとする。

### 新設（DBOBJ）

| メソッド | 役割 | 入力 | 出力 |
|---|---|---|---|
| `in($dir, $tbl)` | テーブル一式（DDL・TSV・keys）の DB 投入。`Psql::in` を呼ぶ | `$dir`, `$tbl` | `$self` |

- `$dir`: テーブル一式の親ディレクトリ。ファイルは `$dir/<name>/` 配下に置かれている
- `$tbl`: テーブル名。`schema.name` 形式を許容し、schema 省略時は `public` とする。`schema.name` 形式のときはディレクトリ名・ファイル名には name 部分のみを使う

### 変更（DBOBJ）

| メソッド | 旧 | 新 |
|---|---|---|
| `psql($sqlfile)` | psql コマンドの組み立て・実行・エラー検知を自前で行う | `DBOBJ::Psql::run($self, '-f', $sqlfile)` を呼ぶ。sqlfile の事前存在チェックは廃止する（development-policy「事前検証の禁止」。ファイル不在は psql 自身が exit=1 で検知し、終了コード経由で die する。確認済み） |
| `spool($spool_id)` | write フェーズまで。SQL の `ORDER BY` を解析してソート列を検証し、`ordercols` を内部状態に保持する | `spool($spool_id, @confirm)`。write フェーズに続けて Spool の確定まで行う。`ORDER BY` 解析・事前チェックは全廃する |
| `prepare($sql)` | sth を作り、`$sql` を内部状態に保持する | sth を作るのみ（SQL 保持は `ORDER BY` 解析専用だったため廃止） |
| `new($dbname)` | 環境変数を検証し、DSN を `%ENV` から直接組む | 検証に加え、`PGHOST`・`PGPORT`・`PGUSER` を内部状態（`host`・`port`・`user`）へ取り込み、DSN・DBI 接続・psql 起動のすべてでこの値を使う |

### 変更なし

`execute`, `run`, `get`, `list`, `arrays`, `hashes`, `close`

### 内部状態の変更

```perl
{
    dbname => $dbname,  # データベース名
    host => $host,    # 接続先ホスト（new() 時に PGHOST から取り込む）
    port => $port,    # 接続先ポート（new() 時に PGPORT から取り込む）
    user => $user,    # 接続ユーザー（new() 時に PGUSER から取り込む）
    dbh => $dbh,      # DBI データベースハンドル
    sth => undef,     # 実行済みステートメントハンドル
}
```

- `sql`・`ordercols` を削除する（いずれも `ORDER BY` 解析専用だった）
- `host`・`port`・`user` を追加する（接続情報の一元化。`new()` 時に環境変数から取り込む）
- `PGPASSWORD` は内部状態に保持しない。DBI へは `new()` の接続時に渡し、psql へは環境変数のまま子プロセスへ引き継ぐ

## 設計方針

### psql 起動の DBOBJ::Psql::run への集約

psql の起動を `DBOBJ::Psql::run($dbo, @args)` に集約する（以下 Psql と略記）。

- 接続情報は `$dbo` が `new()` で取り込んだ値（`dbname`・`host`・`port`・`user`）を使う。Psql は `%ENV` を直接読まない。例外は `PGPASSWORD` のみで、環境変数のまま psql 子プロセスへ引き継ぐ。`PGDATABASE` は参照しない
- `new()` 後に環境変数が変わっても、接続済みオブジェクト経由の psql は `new()` 時の接続情報を使う（DBI と psql の接続先が常に一致する）
- `--set ON_ERROR_STOP=1` で起動する。SQL エラー時に非 0 終了し、NOTICE では終了しない
- 終了コードが 0 以外なら CommonIO の `dying()` でエラーログを残して die する
- `@args` は psql への追加引数（`-f $sqlfile` または `-c $command`）とする
- DDL・keys・SQL ファイルは `'-f'`、`\copy` は `'-c'` でこの関数を呼ぶ

### Psql::in の処理の流れ

1. **schema 分離**: `$tbl` が `schema.name` 形式（`/^(\w+)\.(\w+)$/`）なら schema と name に分離する。形式でなければ schema は `public`、name は `$tbl` のまま。以降、ディレクトリ名・ファイル名には name 部分のみを使う
2. **DDL 実行**: DDL ファイルを次の優先順で決定する
   - `$dir/$name/$name.sql` があればそれを使う
   - なければ `$dir/$name/$name.ddl` を使う
   - どちらも存在しなければ `dying()`
   - 決定した DDL ファイルを `Psql::run('-f', ...)` で実行する（CREATE TABLE 等）
3. **データ投入（`\copy`）— 自動判別**:
   - `$dir/$name/$name.tsv` が存在すれば、それを `\copy $schema.$name from $file` で投入する（単一 TSV）
   - 存在しなければ、`$dir/$name/$name.0000.tsv` から連番（`%04d`）で存在するファイルを順に `\copy` で投入する。番号が途切れたところで終了する（分割 TSV）
   - どちらも存在しなければ投入をスキップする
   - `\copy` は `Psql::run('-c', $command)` で実行する
4. **キー付与**: `$dir/$name/$name.keys` が存在すれば `Psql::run('-f', ...)` で実行する（PRIMARY KEY・UNIQUE・INDEX）。存在しなければスキップする

DBOBJ の `in($dir, $tbl)` は `Psql::in($self, $dir, $tbl)` を呼び、`$self` を返す（メソッドチェーン可能）。

### 実行内容の表示

- 各 psql 実行の前に、実行内容を CommonIO の `log('info', ...)` で表示する（レガシーの STDERR への `print` を置き換える。`log()` は STDERR への出力とログファイルへの保存を兼ねる）
- 表示内容はレガシーの形式を踏襲する: DDL・keys は `sql> \i $file`、データ投入は `sql> copy $schema.$name from $file`

### エラー処理

- DDL ファイルがどちらも存在しない場合は `dying()` で die する
- psql の終了コードが 0 以外の場合は `Psql::run()` 内で `dying()` で die する（sqlfile の不在・DDL の SQL エラー・`\copy` の形式不一致・keys の制約違反はすべてここで捕まる）
- 上記以外の事前検証はしない（development-policy「事前検証の禁止」）
- エラーは一貫して CommonIO の `dying()` でエラーログを残して die する

### TSV の形式

- `\copy ... from $file` のデフォルト形式（タブ区切り・`\N` = NULL）をそのまま使う。Psql 側でオプションは付与しない（レガシー実装と同じ）

### 依存関係

- 新規の外部依存追加なし。DBOBJ::Psql パッケージは CommonIO のみに依存し、DBI を使わない
- ファイル同居のため `use` の追加は不要（`use DBOBJ` で DBOBJ::Psql もロードされる）
- `log()` は `CommonIO::log()` の完全修飾で呼ぶ（import すると Perl 組み込み `log` と衝突するため）

### spool() の確定モード判別

`spool($spool_id, @confirm)` は write フェーズ（`Spool->open` / `add` / `close`）に続けて Spool の確定まで行い、`$spool_id` を返す。確定モードは `@confirm` の形で判別する。

| `@confirm` の形 | 確定 | 意味 |
|---|---|---|
| なし（空） | `Spool::lines($spool_id)` | 行単位の確定。順序不問 |
| 文字列（カラム名）の並び | `Spool::records($spool_id, @confirm)` | 連続同一キーのグループ化。ソート済みが前提 |
| 配列リファレンスの並び | `Spool::grouping($spool_id, @confirm)` | 階層グループの構築 |

- 判別は先頭要素の形のみで行う（配列リファレンスなら `grouping`、文字列なら `records`、空なら `lines`）
- 引数・順序・列名の正しさは事前にチェックしない。schema 外の列名・ソート漏れ（キー再出現）・形の混在は、実行時に Spool / MetaAoh の die が検知する（Spool の confirm は fork 内で行われるため、die は `confirm failed` として伝播する）
- 並び順の指定はグループ指定と別には設けない。`records` のキー列の並び・`grouping` の配列リファレンスに出てきた列の並び（level1…level2…の連結順）を、そのままデータに要求される並び順の前提とする。連続性が保たれていれば昇順・降順は問わない
- `ORDER BY` 解析（`_ordercols`・リテラルマスキング・トップレベル判定）は全廃する
- `prepare()` の SQL 保持・内部状態 `sql`・`ordercols` を削除する

### 実装上の変更点

- `src/DBOBJ.pm` 内に `package DBOBJ::Psql` を新設する（`run`・`in` の2関数。内部関数なし。冒頭に日本語の契約コメントを置く）
- DBOBJ の `psql()` を `DBOBJ::Psql::run` 委譲に置き換え、sqlfile の事前存在チェックを廃止する
- DBOBJ に `in($dir, $tbl)` を新設する（`DBOBJ::Psql::in` 委譲）
- `new()` に `host`・`port`・`user` の取り込みを追加し、DSN・DBI 接続をこの値で組む
- DBOBJ の内部関数 `_psqlrun` は持たない
- DBOBJ の `spool()` を確定モード判別形に書き換え、`_ordercols` と内部状態 `sql`・`ordercols` を削除する
- `_order_spec` は X2Y 規約の `sth2order` に改名する（`hashes()` と `spool()` が schema 規則を共有するための唯一の内部変換関数）。エラーメッセージ専用だった `$api` 引数と冒頭の事前チェック（no column info）を削除し、列情報のない sth は参照時の自然な die に任せる（「事前検証の禁止」）
- `src/DBOBJ.pm` 冒頭の Terms / Rules コメントを design-concept.md の `Rules` と同期する
- README.md・README_ja.md・docs/spec.md・docs/test-spec.md を本変更に合わせて更新する

## テスト方針

- 実 PostgreSQL DB（`develop`）を使う結合テスト（モックなし）
- 環境変数（`PGHOST`, `PGPORT`, `PGUSER`, `PGPASSWORD`）が設定済みであること（定義元は `.claude/settings.json` の `env`）
- テスト用のテーブル一式（DDL・TSV・keys）はテスト内で一時ディレクトリに生成し、テスト後にテーブルを DROP して後始末する
- テストは DBOBJ の公開 API（`psql()`・`in()`・`spool()`）経由で行い、Psql・Spool 連動はその経路で検証する
- 旧 spool テストのうち `ORDER BY` 解析系（旧 #28〜34）は削除し、spool テストを新 I/F で書き直す。全テストを通し番号で振り直す
