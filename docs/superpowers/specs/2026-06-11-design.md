# DBOBJ 設計変更仕様書
Date: 2026-06-11

## 概要

[design-concept.md](../../design/design-concept.md)（2026-06-11 改訂）の To Do「Spool 連携メソッドの新設」を具体化する。

- `spool($spool_id)` を新設する。実行結果を fetch ループで 1 行ずつ Spool へ退避し、`$spool_id` を返す
- `prepare($sql)` が SQL を内部状態に保持するようにする（`spool()` の `ORDER BY` 解析に使う）
- 廃止メソッド一覧から `spool($spool_id)` を削除する（別仕様として復活）

## 変更の背景

- 巨大な結果セットを `hashes()` で metaAoh としてメモリに作るとメモリを使い切る。fetch ループで 1 行ずつ `Spool->open` / `add` / `close` へ流せば、メモリに結果セットを抱えずに退避できる
- 2026-06-10 の設計では「スプールへの書き出しは DBOBJ の責務外」として `spool($spool_id)` を廃止したが、上記のストリーミング退避は「metaAoh を作ってから外で書き出す」方式では代替できないため、別仕様として新設する（[deprecated-methods.md](../../design/deprecated-methods.md) の変更履歴を参照）
- Spool へ流すデータはソート済みが前提（`Spool::records` は再出現で die する）。そのため SQL の `ORDER BY` を確認し、確認できない SQL はエラーに倒す

## API 変更

### 新設

| メソッド | 役割 | 入力 | 出力 |
|---|---|---|---|
| `spool($spool_id)` | 全件を Spool へ退避 | `$spool_id` | `$spool_id` |

### 変更

| メソッド | 旧 | 新 |
|---|---|---|
| `prepare($sql)` | sth を作るのみ | sth を作り、`$sql` を内部状態 `sql` に保持する（外部から見える入出力は変更なし） |

### 変更なし

`new`, `execute`, `run`, `get`, `list`, `arrays`, `hashes`, `psql`, `close`

### 内部状態の変更

```perl
{
    dbname => $dbname,  # データベース名
    dbh => $dbh,      # DBI データベースハンドル
    sth => undef,     # 実行済みステートメントハンドル
    sql => undef,     # 直近に prepare した SQL（spool() の ORDER BY 解析に使う）※新設
    ordercols => [],   # 直近の spool() で解析したソート列名 ※新設
}
```

## 設計方針

### spool() の処理の流れ

1. `prepare()` で保持した SQL のトップレベル `ORDER BY` を解析してソート列を得る（後述）。解析で die する場合、Spool は作らない
2. 解析で得たソート列が SELECT 句の列（`$sth->{NAME}`）にすべて含まれることを確認する。含まれなければ die する
3. ソート列を `ordercols` として内部状態に保持する。呼び出し側は `Spool::records($spool_id, @key_cols)` のキー列としてこれを利用・検証できる
4. `$sth->{NAME}` と DBI の型情報から `hashes()` と同じ規則（`'num'` は `NAME#`、`'str'` は `NAME`）で schema（MetaAoh の order 記法）を組み立て、`Spool->open($spool_id, @order)` を呼ぶ
5. fetch ループで 1 行ずつ取り出し、`undef` を `''` へ置き換えてから `add($row)` する。結果セットを metaAoh や配列としてメモリに作らない
6. fetch 完了後、DBI のエラーを手動検知する（エラーがあれば `dying()`）。`close()` を呼んで `$spool_id` を返す
7. spool の確定（`Spool::records` / `lines` / `grouping`）は呼び出し側の責務であり、`spool()` は write フェーズまでを行う

### ORDER BY 解析

- 対象は SQL のトップレベル（括弧深度 0）にある `ORDER BY` のみ。サブクエリ・ウィンドウ関数など括弧内の `ORDER BY`、文字列リテラル・コメント内の文字列は対象にしない
- 解析は大文字小文字や書式（改行・空白の入り方）の規約に依存しない。規約は保証ではない
- `ORDER BY` 句の終端は、深度 0 の `LIMIT` / `OFFSET` / `FETCH` / `FOR`、または SQL 終端とする
- ソート指定をカンマ（深度 0）で分割し、各指定を次の規則で列名に解決する
  - 裸の識別子（英字または `_` で始まり英数字 `_` が続く）に、任意で `ASC` / `DESC`、`NULLS FIRST` / `NULLS LAST` が付いたもの → 識別子を列名として採用する
  - 上記以外（位置指定 `ORDER BY 1`、関数・演算などの式、`t.col` の修飾名、引用識別子 `"Name"`、`COLLATE`・`USING` 付きなど）→ 「順序を確認できない」として die する
- トップレベルに `ORDER BY` がない SQL は die する
- DBOBJ は渡された SQL を書き換えない。`ORDER BY` の自動付与はしない
- 解析をすり抜けた順序違反（例：別名と実列の不一致）は、Spool 側（`records` の再出現 die）が二段目の安全網として捕まえる

### undef（NULL）の扱い

- 既存取得系と同じ規則で、fetch 時に `undef` を `''` へ置き換えてから `add()` に渡す
- これにより Spool の「`undef` は `add()` で die」契約と両立する

### 0件時の挙動

- 0件でも die しない。`open` → `close` まで行い（Spool の仕様により `close` が `warn` を出す）、`$spool_id` を返す
- 呼び出し側が `Spool::records` で確定すると 0 件（`empty => 1`）で確定する

### エラー処理

- DBOBJ が検知するエラー（SQL 未 prepare・列情報なし・`ORDER BY` 解析失敗・ソート列が SELECT 句にない・fetch 中の DBI エラー）は CommonIO の `dying()` でエラーログを残して die する
- Spool 自身が検知するエラー（spool_id 不正・同一 spool_id の重複・`add()` の検証違反など）は Spool の die をそのまま伝播させる
- `prepare()` を経ていない（内部状態に `sql` がない）状態での `spool()` は die する。`run()` は内部で `prepare()` を呼ぶため、`run()` 経由でも `sql` は保持される

### 依存関係

- `use Spool` を追加する（`lib/Spool.pm`。仕様は `lib/Spool.spec.md`）
- CommonIO・MetaAoh への依存は従来どおり

### 実装上の変更点

- `prepare()` に `$self->{sql} = $sql` を追加する
- `new()` の内部状態初期化に `sql => undef`、`ordercols => []` を追加する
- `spool()` と `ORDER BY` 解析の内部関数を追加する
- `src/DBOBJ.pm` 冒頭の Terms / Rules コメントを design-concept.md の `Rules` と同期する
- README.md・README_ja.md・docs/spec.md・docs/test-spec.md を本変更に合わせて更新する

## テスト方針

- 実 PostgreSQL DB（`develop`）を使う結合テスト（モックなし）
- 環境変数（`PGHOST`, `PGPORT`, `PGUSER`, `PGPASSWORD`）が設定済みであること（定義元は `.claude/settings.json` の `env`）
- テストで作成した spool は `Spool::remove` で後始末する
